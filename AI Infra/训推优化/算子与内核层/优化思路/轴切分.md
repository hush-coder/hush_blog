# 切分的本质

在冯·诺依曼架构下（特别是 GPU/NPU），SRAM 是手动管理的缓存（或者说是暂存池）。

- **为什么切？** 因为 SRAM 很小，一次装不下整个 Tensor。
- **切什么？** 切出刚好能塞进 SRAM 的一块（Tile），算完，存回，再算下一块。
- **核心策略：** 切分的大小和方向，决定了数据复用率和流水线效率。

# 流水（Pipeline）过程

上面是顺序执行，但实际硬件为了提高效率，会让数据搬运（从 DRAM 到 SRAM）和计算（在 SRAM 中进行）同时进行，这就是流水线。常用技术是 双缓冲（Double Buffering）。

## 双缓冲原理

在 SRAM 中开辟两块缓冲区：Buffer0 和 Buffer1。

计算单元使用 Buffer0 中的数据计算时，DMA（数据搬运单元）同时把下一块需要的数据加载到 Buffer1。

计算完 Buffer0 后，立即切换：计算单元使用 Buffer1，同时 DMA 加载下一块到 Buffer0（或者加载更后面的块）。这样计算和访存完全重叠，隐藏了数据加载的延迟。

## 结合 K 维切分的流水步骤
我们以上面的例子，K 切成 4 块（Tile0~Tile3）。假设 SRAM 中有两块缓冲区（BufA/BufB 分别存 A 块和 B 块，为了简化，我们只描述 A 的加载，B 同理，实际需要两个缓冲区分别对应 A 和 B，或者共用）。

下面用文字描述时间步骤（T0, T1, T2...）：

### T0（初始化）：

DMA 开始加载 Tile0 的 A 块（`A[:,0:2]`）和 B 块（`B[0:2,:]`）到 Buffer0。

此时计算单元空闲（等待数据）。

### T1（第一块计算 + 预加载第二块）

- **计算单元**：使用 Buffer0 中的数据，计算，结果累加到 C 的临时累加器（可能在寄存器或 SRAM 另一区域）。
- 同时，DMA 开始加载 Tile1 的 A 块和 B 块到 Buffer1。

> 注意：计算和 DMA 同时进行，互不干扰。

### T2（第二块计算 + 预加载第三块）

计算单元：等待 T1 的 DMA 完成后，立即切换到 Buffer1，使用 Tile1 的数据计算，并累加到 C。

同时，DMA 开始加载 Tile2 的 A 块和 B 块到 Buffer0（此时 Buffer0 已空闲）。

### T3（第三块计算 + 预加载第四块）

计算单元：使用 Buffer0 计算 Tile2。

DMA 加载 Tile3 到 Buffer1。

### T4（第四块计算）

计算单元：使用 Buffer1 计算最后一块。

此时没有下一块需要加载，DMA 空闲。

最终，除了第一次加载（T0）和最后一次计算（T4）无法重叠外，中间的所有计算都与下一次数据加载重叠，大大减少了总时间。

##  字符画时间线
```text
时间轴:  T0        T1        T2        T3        T4
---------------------------------------------------------
DMA:    [加载Tile0] [加载Tile1] [加载Tile2] [加载Tile3] (空闲)
计算:     (空闲)   [算Tile0]  [算Tile1]  [算Tile2]  [算Tile3]
```

可以看到，T1~T3 期间，DMA 和计算单元都在工作，实现了流水。

# 轴切分的选择策略

## 张量的维度

假设我们处理的是最常见的图片数据（比如一张彩色照片）。在深度学习模型中，图片通常被表示成一个四维张量，形状是 [N, C, H, W]：

- **N（Batch，批次大小）**： 一次处理多少张图片。比如 N=32，就表示一次同时处理 32 张不同的照片。
- **C（Channel，通道数）**： 图片的颜色通道。彩色图片通常是红、绿、蓝三个通道，所以 C=3。黑白照片只有一个通道，C=1。
- **H（Height，高度）**： 图片的高，比如 224 像素。
- **W（Width，宽度）**： 图片的宽，比如 224 像素。

除了图片，还有全连接层（也就是矩阵乘法）里的数据。它的形状可能是 [M, K] 或 [K, N]，你可以理解为：

- **M（可以想象成批次或输出神经元数）**： 一批数据里有多少个样本，或者这一层有多少个输出神经元。
- **K（可以想象成输入特征数）**： 每个样本有多少个输入特征。
- **N（可以想象成输出特征数）**： 这一层要计算出多少个输出结果。

## 矩阵乘法（比如全连接层）的切分

想象你要计算 C = A * B。A 是一个大矩阵（比如 M×K），B 是另一个大矩阵（K×N）。计算 C 的每一个元素，都需要 A 的一行和 B 的一列相乘。

### M 维度（可以理解成“样本数”或“输出行的数量”）

如果沿着 M 切，就是每次只计算 A 的几行。这意味着这几行对应的所有输出（C 的几行）都会在工作台上算完。

- **优点**：B 矩阵可以被这几行反复使用，只要把 B 整个放在工作台上，就能减少从内存读 B 的次数。
- **缺点**：如果 M 很大，而 B 也很大，工作台可能放不下整个 B。而且如果 B 的某些部分只被用一次就丢掉，浪费了。

### N 维度（可以理解成“输出列的数量”）

如果沿着 N 切，就是每次只计算 C 的几列。这意味着每次只需要 B 的几列，A 可以放在工作台上反复用。

- **优点**：A 可以被复用。
- **缺点**：同理，如果 A 太大，工作台放不下。

### K 维度（可以理解成“输入特征的维度”，也就是求和的那个维度）

这是最常见的切法。我们不是一次性算出 C 的完整值，而是先计算“部分和”。比如把 K 切成 4 块，先算第一块 K，得到 C 的一部分结果（累加值），存到工作台上；然后加载第二块 K，继续累加……最后得到完整的 C。

***为什么选 K 维？***

K 维度是 A 的列数，也是 B 的行数。它代表的是“中间求和”的维度。

- 如果你切分 M 或 N，你是在切分最终菜品的“行”或“列”，但每次仍然需要把整个 K 维度（所有食材和所有调味料）搬上工作台，工作台还是放不下。
- 切分 K 就是把长长的 K 切成若干小段（比如 K1, K2, K3），每次只处理一小段 K。这样每次工作台上只需要放：
    - A 的一整行（或几行）中对应这一小段 K 的部分（一小段食材）
    - B 的一整列（或几列）中对应这一小段 K 的部分（一小段调味料）这样工作台就能放下了。

更重要的是：**每次算出的结果是“部分和”**。例如第一次算 K1 段，得到一个中间结果；第二次算 K2 段，把两次结果累加起来，最后得到完整的C。这就把一个大问题拆解成了多个可以塞进 SRAM 的小问题。


## 逐元素操作的切分

逐元素操作，比如给图片的每个像素都加一个数，或者用 ReLU 激活函数（把负数变成 0）。这种计算很简单，就是每个数据自己算自己的，互不影响。

- N 维（批次）：一次只处理几张图片，而不是全部 32 张。
- H 维（高度）：一次只处理图片的上面几行，而不是整张图。
- W 维（宽度）：一次只处理图片的左边几列。

***选择策略：***

通常会选择 H 或 N 维 来切分。为什么？

- **原因 1：保持连续性。**内存里数据的排列通常是先存完一张图的所有像素（先 H 后 W），再存下一张。如果切 W 维，可能会把一行数据切成两半，导致访问内存时效率变低（就像读书时不能一口气读完一行，要来回翻页）。切 H 维，则是按行切，每一块的数据在内存里是连续的一大段，读起来快。
- **原因 2：负载均衡。**切 N 维，让每个计算单元处理不同数量的图片，很容易分配均匀。切 H 维也是类似，让每个单元处理差不多行数的图片。
- **简单比喻：**你有一本很厚的书（张量），要每页都盖个章（逐元素操作）。工作台太小，不能摊开整本书。你会怎么切？你可能会把书按页（N）切分，一次只拿几页；或者按段落（H）切分，一次只拿几段。你不会把一行字（W）从中间撕开，因为那样读起来麻烦。

# 考虑因素

切分不仅仅是“把数据掰开”，而是需要权衡以下几个关键点：

## 1. 流水线气泡

这是流水计算中最需要考虑的问题。

理想情况： 计算单元在算 Tile 0 的时候，DMA（数据搬运单元）已经把 Tile 1 从 DRAM 搬到了 SRAM（Double Buffering）。

问题： 如果切分的块（Tile）太小，传输次数变多，每次传输的启动开销（Latency）占主导，流水线会被“灌不满”，导致计算单元闲置等待数据。

权衡： 需要在 SRAM 容量允许的情况下，尽可能切大块，减少切分的片数，让传输时间能够被计算时间完全掩盖。

## 2. 局部性

为什么要在某个轴上切分？

数据复用： 切分是为了让数据能在 SRAM 里多待一会儿。

例子： 在做矩阵乘法时，切分一个子矩阵放在 SRAM 里，这个子矩阵会被反复使用很多次，然后再换下一块。如果切错了轴，导致加载的数据只用一次就被踢出，那就没有意义了。

## 3. 切分粒度

细粒度： 更灵活的调度，但对 SRAM 的分配和管理要求高，容易产生碎片或 Bank Conflict。

粗粒度： 管理简单，流水线容易填满，但可能因为 SRAM 装不下导致切分失败。

## 4. 边界处理

当 Tensor 的维度无法被 Tile 大小整除时，会留下一些残余（Residue）。

这些残余边角的计算逻辑往往比较复杂，要么用特殊的 Kernel 处理，要么用 Mask 掩盖掉多余的计算。设计切分策略时，要尽量减少这种残余带来的控制流分歧。

## 5. 计算强度

访存受限任务： 如果计算很简单（比如逐元素操作），切分的目标很简单，就是把数据喂饱，尽量让访存带宽跑满。

计算受限任务： 如果计算很复杂（比如大矩阵乘），切分的目标是把计算单元喂饱，要尽量让数据在 SRAM 里待久一点，少去 DRAM 读数据。
