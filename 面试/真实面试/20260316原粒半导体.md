# 1. 算子

## 1.1 轴切分搬运时的考虑因素（算子）

在AI编译器和算子设计中，轴切分搬运是指将张量数据沿特定维度切分为更小的块（Tile），并在内存层次结构（如全局内存->共享内存->寄存器）中进行搬运。这一过程直接决定算子的访存效率和计算吞吐。以下是核心考虑因素：

### 1.1.1 内存层次结构与带宽特性

HBM/全局内存：容量大、带宽有限（~TB/s），需合并访问（Coalesced Access）并减少事务数。切分后的数据块应连续存储，使线程束的访问能合并为少数缓存行。

共享内存/L1缓存：带宽高（~10TB/s）、容量小（~几十KB），切分大小需适配其容量，避免溢出导致寄存器溢出或多次搬运。同时需考虑bank冲突：切分后的访问模式应尽量使同一warp内线程访问不同bank（如按列切分时，步长应为奇数或采用padding）。

寄存器文件：私有但极高速，切分粒度过细会导致寄存器压力过大，降低occupancy；过粗则可能无法掩盖访存延迟。

### 1.1.2 切分轴与数据复用

复用机会：选择切分轴时需识别数据重用的模式。例如卷积中，输入通道维（C）切分后，可复用同一输入像素到多个输出通道；输出通道维切分则可复用权重。编译器通过分析数据流图，选择复用度高的轴优先切分，减少搬运次数。

切分顺序：优先切分最内层循环（如计算密集的维度），使得内层计算完全在高速缓存中完成，避免反复从低层内存搬运。例如在矩阵乘法中，将K维（累加维）切分到共享内存中循环累加，使输入分块驻留共享内存。

### 1.1.3 并行粒度与硬件资源
线程块/线程束级切分：在GPU上，切分通常以CTA（Cooperative Thread Array）为单位，每个CTA处理一个输出tile。切分大小需平衡：太大可能导致CTA资源不足（共享内存、寄存器），限制并行度；太小则无法隐藏内存延迟。

warp级切分：在warp内进一步切分，可利用warp shuffle指令高效通信，避免共享内存。例如在Reduce操作中，可在warp内先部分规约，减少全局内存访问。

SIMD宽度：CPU上的向量化（AVX）要求切分后数据对齐且连续，便于向量加载。

### 1.1.4 数据搬运与计算重叠
双缓冲（Double Buffering）：将搬运和计算流水化。例如切分后，一个buffer用于当前计算，另一个buffer预取下一块数据。切分大小需保证计算时间大于或等于搬运时间，以达到掩盖延迟。

异步拷贝：利用硬件特性（如GPU的异步拷贝指令cp.async）实现搬运与计算并行。切分块数应足够多以保持流水线充满。

同步开销：切分边界可能引入同步（如__syncthreads）。尽量减少同步点，例如通过调整切分形状使同一warp内无需同步，或使用无同步的算法（如warp级规约）。

### 1.1.5 切分形状与访存模式
Bank冲突避免：共享内存按bank组织，切分时若多个线程访问同一bank的不同地址（如按列访问二维数组，列数为2的幂次），会导致冲突。可通过padding或改变切分形状（如使用float4向量化访问）来缓解。

向量化加载/存储：切分后数据尽量满足向量化宽度（如128-bit），使用ldg128指令提升带宽利用率。例如切分通道数为4的倍数，便于向量访存。

非规则切分：对于稀疏或变长数据，需考虑动态切分策略，可能引入索引重计算或预取开销。

### 1.1.6 自动调度与代价模型
机器学习驱动的切分搜索：AI编译器（如Ansor、AutoTVM）通过代价模型预测不同切分方案的性能，考虑因素包括：计算密度（FLOP/byte）、缓存命中率、并行空闲率等。

启发式规则：如优先切分最内层维，避免跨步访问；若数据量小于缓存，则不切分；若存在广播操作，切分广播源维度以减少冗余搬运。

### 1.1.7 硬件特性适配
GPU张量核（Tensor Core）：切分需对齐到张量核的输入格式（如16x16的片段），并要求数据布局（如NC1HWC0）以支持快速加载。

TPU脉动阵列：切分需匹配矩阵分块大小（如128x128），并考虑数据在HBM和向量内存间的搬运模式。

NUMA架构：多CPU场景下，切分需考虑跨socket访存延迟，尽量保持数据本地性。

### 示例：矩阵乘法优化中的切分搬运
以CUDA中优化GEMM为例：

将矩阵A、B切分为BM×BK和BK×BN的Tile，加载到共享内存。

切分轴选择：K维（累加维）切分，使每个Tile完全在共享内存中累加，避免反复从全局内存读取A/B。

搬运考虑：使用向量化加载（float4）合并访问全局内存，双缓冲技术预取下一Tile，同时通过调整BM、BN平衡寄存器压力与occupancy。

同步：每加载完一个Tile后需同步，但可优化为warp级部分计算以减少同步次数。

### 总结

总结：轴切分搬运的本质是在硬件资源约束下，通过合理划分数据块，最大化数据重用并隐藏访存延迟。编译器需结合内存层次、并行模型和硬件特性，自动或启发式地选择切分轴、块大小和搬运策略，以逼近理论峰值性能。

## 1.3 conv算子的计算过程（算子）

**卷积算子的本质是滑窗乘加运算**。我们有一个输入特征图（形状 NCHW）和一组卷积核（KCHW），输出特征图每个位置的值是通过将卷积核与输入对应窗口做点积得到的。最直接的实现就是多层循环：遍历每个输出位置，对于每个输出通道，累加所有输入通道在滑窗内的乘积累加。但实际中不会这样直接写，因为访存效率极低。

为了提高性能，现代算子库通常采用以下几种方法：

- **im2col + GEMM**：把输入中每个滑窗区域拉成一列，构成一个大矩阵，同时把卷积核也展开成矩阵，然后调用高度优化的通用矩阵乘法（GEMM）库。这样做的好处是利用了矩阵乘的极致优化，但代价是显式展开数据会占用额外内存，且数据重排有开销。
- **Winograd 卷积**：针对 3x3 等小卷积核，通过数学变换减少乘法次数。例如 F(2x2, 3x3) 将输入 tile 和卷积核变换到另一个空间，做元素级乘法再变换回来，乘法次数从 36 次降到 16 次左右，显著提升计算速度。但变换本身有额外计算，适合计算密集场景。
- **FFT 卷积**：利用时域卷积等于频域相乘，对输入和核做 FFT，频域相乘后再 IFFT 回来。适合大卷积核，但小核时不划算。

在 AI 编译器和算子开发层面，我们更关心如何切分数据、利用内存层次结构。比如在 GPU 上实现卷积时，会考虑：

- 数据布局选择（NHWC 还是 NCHW）影响访存连续性；
- 将输出通道、输入通道、空间维度分块，让数据尽量留在共享内存或寄存器中；
- 利用向量化加载（如 float4）和双缓冲技术掩盖访存延迟；1
对于支持 `Tensor Core` 的 GPU，还会将数据排布成特定格式，调用硬件加速指令。

最后，编译器的任务就是根据硬件特性和输入形状，自动搜索或选择最优的实现方式（比如是 im2col 还是 Winograd），生成高性能代码。

## 1.5 搬运的中间缓存大小设置（算子）

### 1.5.1. 基本思想：计算时间 ≥ 数据加载时间
双缓冲需要让一个缓冲区在进行计算的同时，另一个缓冲区在加载下一块数据。理想情况下，计算当前块的时间应该大于或等于加载下一块的时间，这样流水线才能持续饱满，不会出现计算单元等待数据的情况。

计算当前块所需时间 ≈ 块数据量 / 有效带宽（包括 PCIe、HBM 带宽，以及可能的同步开销）

大于

加载下一块数据所需时间 ≈ 块内的计算量 / 计算吞吐（受硬件 SM 数量、频率、指令吞吐等影响）

这两个时间都与分块大小直接相关：

因此，我们可以通过调整块大小使二者匹配。如果块太小，计算太快完成，流水线会因等待加载而停顿；如果块太大，加载时间过长，计算可能已经完成，同样造成等待。

### 1.5.2. 硬件资源约束
分块大小不能无限增大，必须受限于片上存储（如 GPU 的共享内存、L1 缓存、寄存器）的大小：

- 共享内存：两个缓冲区通常需要同时驻留在共享内存中（一个用于当前计算，一个用于预取），所以块大小必须 ≤ 共享内存容量 / 2（还要考虑其他开销）。
- 寄存器：如果分块导致每个线程的寄存器压力过大，会降低 occupancy，反而可能拖慢计算。
- 线程块数量：分块过大可能使线程块占用的资源过多，导致并行线程块数减少，降低硬件利用率。

因此，分块大小往往是在资源限制的上限附近，再结合流水线平衡进行微调。

### 1.5.3 并行粒度与延迟隐藏

分块大小也影响线程级的并行度。例如在 GPU 上，一个线程块处理一个输出分块。如果分块太小，线程块数量多，但每个块计算量少，可能无法充分隐藏内存延迟（因为线程块切换可以隐藏延迟，但块内计算太短会导致频繁切换开销）。反之，分块太大可能导致块内线程数过多，同步开销增加。

### 1.5.4 访存模式与连续性

分块形状也会影响实际的数据传输效率。比如在矩阵乘法中，从全局内存加载 A 的分块时，我们希望访存是合并的（coalesced），这就要求分块的布局与内存布局匹配。同时，分块大小应尽量对齐到缓存行或向量化宽度（如 128 字节），以提高带宽利用率。

# 2. C++

## 2.1 C++的左值右值、移动语义、move、border（C++）

### 2.1.1 左值和右值：表达式的值类别

C++ 中的每个表达式都有两个属性：**类型**和**值类别**。值类别是 C++11 引入的精细分类，但核心仍围绕左值和右值。

- **左值（lvalue）**：指的是有持久地址的表达式，可以取地址，通常是一个变量或对象。例如：

```cpp
int x = 10;   // x 是左值
int* p = &x;  // 可以对 x 取地址
```

- **右值（rvalue**）：指的是临时对象或字面量，没有持久地址，不能被取地址。右值又分为纯右值（prvalue，如 42、x+y）和将亡值（xvalue，如 std::move(x) 的结果）。简单理解：右值就是“即将销毁的值”，其资源可以被“窃取”。例如：

```cpp
int&& r = 42;   // 42 是右值
int y = x + 5;  // x+5 的结果是右值
```

> **区分**：能取地址的是左值，否则是右值。

### 2.1.2 右值引用（T&&）—— 绑定到右值的引用

右值引用是一种只能绑定到右值的引用类型，用 && 声明。它延长了临时对象的生命周期，并允许我们修改那个临时对象（这是关键，因为左值引用 const 也能绑定右值，但不能修改）。

```cpp
int&& rr = 42;        // OK，rr 绑定到临时 42
rr = 100;             // 可以修改
```
右值引用的主要目的是支持移动语义：当我们有一个即将销毁的右值对象时，我们可以“偷走”它的资源（如堆内存），而不是深拷贝。

### 2.1.3 移动语义：移动构造函数和移动赋值运算符

移动语义允许一个对象将资源转移给另一个对象，而无需拷贝。典型应用是管理动态内存的类（如 std::vector、std::string）。

- **移动构造函数**：T(T&& other)
从 other 中“窃取”资源（如指针），然后将 other 置为空，避免拷贝。
- **移动赋值运算符**：T& operator=(T&& other)

**示例：**

```cpp
class MyString {
    char* data;
public:
    // 移动构造函数
    MyString(MyString&& other) noexcept 
        : data(other.data) {
        other.data = nullptr;   // 源对象不再拥有资源
    }
    // 拷贝构造函数（深拷贝）
    MyString(const MyString& other) {
        data = new char[...];   // 复制数据
    }
};
```
> 当用一个右值初始化对象时，编译器会选择移动构造函数而不是拷贝构造函数，从而大幅提升性能。

### 2.1.4 `std::move` —— 将左值强制转换为右值引用

std::move 本质上只是一个强制类型转换：`static_cast<T&&>(obj)`。它告诉编译器：“这个对象可以被移动”。但它并不移动任何东西，只是改变了值类别，使得后续可以匹配移动构造函数。

***为什么需要它？***

有时候我们有一个左值，但我们知道它的生命周期即将结束，想要移动它而不是拷贝。例如：

```cpp
MyString s1("hello");
MyString s2 = std::move(s1);   // s1 被转为右值，调用移动构造
// 之后 s1 处于“有效但未指定”状态，通常为空
```
> 注意：使用 `std::move` 后，不应再使用原对象的值，除非重新赋值。

### 2.1.5 完美转发和forward

***转发引用（万能引用）***

在模板中，如果参数类型写为 `T&&`，它并不一定表示右值引用，而是转发引用（也称为万能引用），其具体含义取决于模板参数 T 的推导。

```cpp
template<typename T>
void wrapper(T&& arg) {  // arg 是转发引用
    // ...
}
```

- 当传入一个左值（如 `int x`）时，T 被推导为 `int&`，根据引用折叠规则，`T&&` 变为 `int& &&`，折叠为 `int&`，因此 `arg` 的类型是 int&（左值引用）。
- 当传入一个右值（如 42）时，T 被推导为 int，`T&&` 就是 `int&&`（右值引用）。

这样，`wrapper` 可以同时接受左值和右值，并且 arg 的类型会精确匹配传入参数的值类别（左值引用或右值引用）。但问题在于，在函数内部，arg 本身是一个变量，无论其类型是左值引用还是右值引用，它都是一个左值表达式（因为它有名字和地址）。如果直接调用 `foo(arg)`，那么 foo 始终看到的是左值。为了恢复 arg 原本的值类别，我们需要 `std::forward`。

***`std::forward` 的工作原理***

`std::forward` 是一个条件转换函数，定义在 <utility> 中。它的典型实现如下（简化版）：

```cpp
template<typename T>
T&& forward(typename std::remove_reference<T>::type& arg) noexcept {
    return static_cast<T&&>(arg);
}

template<typename T>
T&& forward(typename std::remove_reference<T>::type&& arg) noexcept {
    static_assert(!std::is_lvalue_reference<T>::value, "bad forward");
    return static_cast<T&&>(arg);
}
```

实际上，标准库通常只提供一个版本，利用引用折叠规则来实现。核心在于：当 T 被推导为左值引用时，`forward<T>` 返回左值引用；当 T 被推导为非引用时，返回右值引用。

关键点：`std::forward` 需要显式指定模板参数 T，而这个 T 正是转发引用被推导出来的类型。因此，完美转发的标准写法是：

```cpp
template<typename T>
void wrapper(T&& arg) {
    foo(std::forward<T>(arg));   // 将 arg 按原始值类别转发
}
```

- 如果传入的是左值 `int x`，则 T 为 `int&`，`std::forward<int&>(arg)` 返回左值引用，foo 收到左值。
- 如果传入的是右值 42，则 T 为 int，`std::forward<int>(arg)` 返回 `int&&`，foo 收到右值，从而可以触发移动语义。

### 2.1.6 移动构造函数

移动构造函数是一种特殊的构造函数，它接受一个右值引用参数，从源对象“窃取”资源（如堆内存、文件句柄等），并将源对象置于有效但未指定的状态（通常是空状态）。这样避免了昂贵的深拷贝操作。

```cpp
class String {
    char* data;
    size_t size;
public:
    // 拷贝构造函数（深拷贝）
    String(const String& other) {
        data = new char[other.size];
        memcpy(data, other.data, other.size);
        size = other.size;
    }
    
    // 析构函数
    ~String() { delete[] data; }
};
```
***关键点：***

- **窃取资源**：直接将源对象的指针赋值给自己
- **置空源对象**：防止源对象析构时释放已转移的资源
- **noexcept**：移动操作通常不应抛出异常，以便标准库优化（如 vector 重新分配时优先使用移动而非拷贝）

***移动构造函数在以下情况被调用***

1. 用右值初始化对象：

```cpp
String s1("hello");
String s2 = std::move(s1);  // 调用移动构造（s1 被转为右值）
```

2. 函数返回局部对象（返回值优化失效时）：

```cpp
String createString() {
    String temp("hello");
    return temp;  // 这里可能发生移动构造（或 RVO）
}
```

3. 标准库容器操作：

```cpp
std::vector<String> v;
v.push_back(String("hello"));  // 移动构造临时对象

String s;
v.push_back(std::move(s));     // 移动构造 s
```

4. 函数参数传递（按值传递时）：

```cpp
void takeString(String s) { ... }

String str;
takeString(std::move(str));  // 调用移动构造
```

## 2.2 C++的多态

C++ 的多态（Polymorphism）是指同一接口（函数名或操作）在不同类型的对象上表现出不同的行为。它是面向对象编程的核心特性之一，使得代码更灵活、可扩展。C++ 中的多态主要分为静态多态（编译时多态）和动态多态（运行时多态）。

### 2.2.1 静态多态（编译时多态）

在编译阶段就确定具体调用哪个函数，通过**函数重载**、**运算符重载**和**模板**实现。

- **函数重载**：同一作用域内多个同名函数，参数列表不同（个数、类型、顺序），编译器根据实参类型选择匹配版本。

```cpp
void print(int i);
void print(double d);
void print(const char* s);
```

- **运算符重载**：为自定义类型赋予运算符特定含义，本质也是函数重载。
- **模板**：包括函数模板和类模板。编译器根据模板实参推导生成具体代码（模板实例化），实现静多态，例如 STL 容器和算法广泛使用模板。

静态多态的优点是没有运行时开销，调用在编译期确定；缺点是灵活性较低，代码必须完全确定。

### 2.2.2 动态多态（运行时多态）

在程序运行时根据对象的实际类型决定调用哪个函数，通过继承和虚函数实现。

***核心机制***

- **虚函数**：基类中使用 `virtual` 关键字声明的成员函数，允许派生类重写（`override`）。
- **虚函数表（vtable）**：每个包含虚函数的类都有一个虚函数表，表中存放该类的虚函数地址。
- **虚指针（vptr）**：每个对象内部有一个隐藏指针（`vptr`），指向所属类的虚函数表。

当通过基类指针或引用调用虚函数时，程序会**动态绑定**：从对象的 `vptr` 找到 `vtable`，再从 `vtable` 中取出正确的函数地址执行。

***示例***

```cpp
class Animal {
public:
    virtual void speak() const { cout << "Animal speaks\n"; }
    virtual ~Animal() {}  // 虚析构函数，确保正确释放派生类对象
};

class Dog : public Animal {
public:
    void speak() const override { cout << "Dog barks\n"; }
};

class Cat : public Animal {
public:
    void speak() const override { cout << "Cat meows\n"; }
};

void makeSound(const Animal& a) {
    a.speak();  // 运行时根据实际对象类型调用
}

int main() {
    Dog d;
    Cat c;
    makeSound(d);  // 输出 "Dog barks"
    makeSound(c);  // 输出 "Cat meows"
}
```

***条件***

1. 存在继承关系。
2. 基类中声明虚函数，派生类中重写（override）。
3. 通过基类的指针或引用调用虚函数。

***注意事项***

- **虚析构函数**：如果基类析构函数不是虚函数，删除基类指针指向的派生类对象时，只会调用基类析构函数，导致派生类资源泄漏。
- **纯虚函数与抽象类**：包含纯虚函数（如 virtual void f() = 0;）的类称为抽象类，不能实例化，用于定义接口。
- **性能开销**：虚函数调用比普通函数多一次间接寻址（通过 vptr 和 vtable），且通常无法内联。但现代编译器能进行一定优化。
- **覆盖（override）与隐藏（hide）**：派生类中如果函数名与基类相同但参数不同，会隐藏基类同名函数，而非覆盖。C++11 引入 override 关键字显式标明意图，避免意外隐藏。

### 2.2.3 多态的应用与意义

- **解耦与扩展**：通过基类接口编程，新增派生类无需修改原有代码（开闭原则）。
- **设计模式实现**：如工厂模式、策略模式、观察者模式等大量依赖多态。
- **框架设计**：如 GUI 事件处理、游戏引擎中角色行为定义。

### 2.2.4 总结

| 特性 | 静态多态（编译时） | 动态多态（运行时） |
| :--- | :--- | :--- |
| 绑定时机 | 编译期 | 运行期 |
| 实现方式 | 重载、模板 | 虚函数、继承 |
| 速度 | 快（无间接调用） | 略慢（一次间接寻址） |
| 灵活性 | 较低（类型必须编译时确定） | 高（运行时决定类型） |
| 典型应用 | 泛型编程（STL） | 面向对象接口、插件化架构 |

C++ 多态通过静态和动态两种形式，既保留了编译时的高效，又提供了运行时的灵活。理解虚函数表机制和正确使用虚析构函数，是编写健壮可扩展 C++ 程序的基础。在实际开发中，根据场景选择合适的多态方式：需要极致性能且类型固定时用模板；需要运行时多态和接口抽象时用虚函数。
